Anatomy and Verification of a JWT

JSON Web Tokens (JWTs) are the most popular mechanism for stateless authentication. To understand how third-party services like Clerk or Firebase secure your APIs, you must understand exactly how a JWT is structured and verified.

What is a JSON Web Token?

A JWT is a compact, URL-safe means of representing claims (pieces of information) to be transferred between two parties. It is defined in RFC 7519.

A JWT is composed of three parts, separated by periods (.):

The Three Parts Explained

1. The Header

The header typically consists of two parts: the type of the token (JWT) and the signing algorithm being used, such as HMAC SHA256 (HS256) or RSA (RS256).

{
  "alg": "HS256",
  "typ": "JWT"
}

This JSON is Base64Url encoded to form the first part of the JWT.

2. The Payload

The payload contains the **claims**. Claims are statements about an entity (typically, the user) and additional metadata. There are three types of claims: registered, public, and private claims.

  • Registered claims: Predefined standard claims like sub (subject/user ID), exp (expiration time), iat (issued at time), and iss (issuer).
  • Private claims: Custom claims created to share information between parties (e.g. role or tenant_id).
{
  "sub": "1234567890",
  "name": "John Doe",
  "role": "ADMIN",
  "exp": 1782384000
}

This JSON is Base64Url encoded to form the second part of the JWT. Crucial reminder: Base64Url is encoding, NOT encryption. Anyone can decode and read this payload. Never store secrets inside a JWT payload.

3. The Signature

To create the signature, you take the encoded header, the encoded payload, a secret key, and hash them using the algorithm specified in the header.

HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret_key
)

The signature is used to verify that the sender of the JWT is who it says it is and to ensure that the message wasn't changed along the way.

Cryptographic Signing: Symmetric vs. Asymmetric

How the signature is verified depends on the cryptographic scheme used:

Symmetric: HS256 (HMAC with SHA-256)

Both the issuer (auth server) and the consumer (your API server) share a single secret key. The auth server signs the token with the secret, and your API server verifies it with the exact same secret. Simple, but require secure key distribution.

Asymmetric: RS256 (RSA Signature with SHA-256)

The auth provider (like Clerk, Auth0, or InsForge) has a private key (secret) to sign the token. They publish their public key (available via a public JSON Web Key Set - JWKS endpoint). Your application server downloads this public key and uses it to verify the signature. Your server doesn't know the private key, making this extremely secure.

How Verification Works Step-by-Step

When your Next.js/Node.js backend receives a request with an Authorization: Bearer <token> header, it performs these verification steps:

  1. Parse: Splits the token by . into Header, Payload, and Signature.
  2. Algorithm Check: Reads the header to identify the algorithm. Vulnerability Alert: Always reject the "alg": "none" header variant to prevent validation bypasses.
  3. Signature Verification: Re-calculates the signature using the received Header, Payload, and the known verification key. It compares this calculation to the signature received in the token. If they differ, validation fails.
  4. Expiration Check: Decodes the payload, parses the exp claim, and rejects the token if the current system time is past the expiration time.
