OAuth 2.0 and OpenID Connect (OIDC)

When users choose to "Log in with Google" or when your backend talks to third-party identity management providers (like Clerk), you are using OAuth 2.0 and OpenID Connect (OIDC) under the hood. Understanding these protocols is essential for building modern SaaS integrations.

Authentication vs. Authorization

Although they are often grouped together, these two terms mean different things:

  • Authentication (AuthN) is the process of verifying who a user is ("Are you John Doe?").
  • Authorization (AuthZ) is the process of verifying what a user can do ("Can John Doe edit this task?").

Historically, **OAuth 2.0** was designed strictly as an **Authorization** framework. It issues an access_token so a client application can perform actions on behalf of a user. It does not provide profile details or say how the user authenticated.

**OpenID Connect (OIDC)** is a simple identity layer built **on top of OAuth 2.0**. It introduces an id_token (specifically a JWT) that contains standardized user profile attributes, transforming OAuth 2.0 into an **Authentication** protocol.

The OAuth 2.0 Roles

Every flow involves four distinct actors:

  1. Resource Owner: The user granting access to their account.
  2. Client: The application requesting access (e.g., ProjectForge).
  3. Authorization Server: The server managing authentication, consent, and token issuing (e.g., Clerk, Google Identity).
  4. Resource Server: The API hosting the protected user data (e.g., GitHub API, Google Calendar API).

The Modern Standard: Auth Code Flow with PKCE

For modern web applications, Single Page Applications (SPAs), and mobile apps, the industry standard is the **Authorization Code Flow with PKCE (Proof Key for Code Exchange)**. PKCE prevents interception of authorization codes in public clients.

Auth Code Flow with PKCE

How PKCE Secures the Flow:

  1. Code Verifier: The client generates a high-entropy cryptographically random string (the verifier).
  2. Code Challenge: The client hashes the verifier using SHA-256 and Base64Url encodes it. This challenge is sent during Step 1.
  3. Verification: During the code-for-token exchange (Step 5), the client sends the raw **Code Verifier**. The Auth Server hashes this verifier and checks if it matches the original **Code Challenge**. If an attacker had intercepted the Auth Code, they would still lack the Code Verifier, preventing token issuance.

The Three Issued Tokens

  • ID Token (OIDC): A JWT containing user profile info (name, email, profile picture) used to log the user in on the client side.
  • Access Token: An opaque string or a JWT used by the client to make authorized requests to the Resource Server (API).
  • Refresh Token: A long-lived credential used to request new Access Tokens once they expire, avoiding prompt-logins.
Quiz: Test Your Storage Strength
1. What is the primary purpose of OpenID Connect (OIDC) compared to OAuth 2.0?
Correct! OIDC builds a standardized identity profile layer (using ID tokens) on top of the authorization structures established by OAuth 2.0.
Incorrect. Recall the definition. OAuth 2.0 alone is only for authorization (giving permissions). OIDC adds the "who is this user" identity piece.
2. Why was PKCE (Proof Key for Code Exchange) added to the Authorization Code Flow?
Correct! By requiring the secret Code Verifier during token exchange, PKCE ensures that an intercepted authorization code alone cannot be swapped for access tokens.
Incorrect. Think about what we said about public clients (like SPAs and mobile apps) where authentication codes might be exposed in browser history or OS logs.
