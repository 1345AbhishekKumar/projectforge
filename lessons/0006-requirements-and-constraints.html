Engineering Requirements & Constraints

Before writing a single line of code for a complex system like ProjectForge, you must convert ambiguous human goals into concrete, testable technical parameters. This process splits design inputs into Requirements and Constraints.

The Anatomy of Engineering Inputs

When stakeholders say, "We need a fast, secure task manager that complies with privacy laws and runs in the cloud," a software architect hears a mixture of functional behaviors, quality attributes, and rigid limitations. We categorize these into three distinct buckets:

1. Functional Requirements (FR)

What the system **must do**. These describe the specific features, capabilities, and behaviors of the application. They have direct business value and represent user actions.

  • Example: "The system must allow users to invite members to their organization via an email invitation."
  • Verification: Tested using integration or End-to-End (E2E) tests that simulate user actions (e.g., inviting a user and verifying they receive the invite link).

2. Non-Functional Requirements (NFR)

How the system **must behave**. NFRs define quality attributes, performance envelopes, availability levels, and security standards. They are often called the "-ilities" (scalability, reliability, security, availability).

  • Example: "The system must load the workspace dashboard in under 1 second under ordinary load."
  • Verification: Measured using load testing tools, performance profiling, and browser auditing (like Lighthouse).

3. Constraints

The **boundaries you cannot cross**. Unlike requirements (which you build the system to satisfy), constraints are pre-existing limitations imposed by business, technology, time, budget, or law. You do not design constraints; you design **around** them.

  • Example: "The system must execute on the NextJS web host (Vercel) using PostgreSQL."
  • Verification: Enforced by architecture audits and infrastructure configuration. If you violate a constraint, the system cannot be deployed or is legally invalid.

Requirements vs. Constraints: The Litmus Test

It is common to confuse NFRs and Constraints. To tell them apart, ask yourself: "Can I trade this off for better performance or lower cost?"

  • Non-Functional Requirements: Can be negotiated. You can choose to target a 1.5-second load time instead of 1.0 second to reduce server costs.
  • Constraints: Are non-negotiable. If the law says user data must reside in the European Union (GDPR), you cannot "trade off" compliance to save database latency. You must build within that geographic boundary.

Desirable Difficulty Tip: When drafting requirements for ProjectForge, write them as verifiable statements.
