Lesson 0007 • Domain Design

Domain Modeling & Bounded Contexts

In complex enterprise platforms like ProjectForge, your database schemas and code interfaces can quickly devolve into a tangled "big ball of mud" if not organized correctly. Domain-Driven Design (DDD) provides the architectural framework to build clean, modular domain layers.

Ubiquitous Language & Bounded Contexts

The core philosophy of DDD is that software code must reflect a shared mental model of the business. Two concepts establish this model:

  • Ubiquitous Language: A common, rigorous language co-created by developers and business stakeholders. If a PM calls an item a "Task" and a developer calls it a "Ticket" in the database, the language is broken. In the codebase, it must be consistently named Task.
  • Bounded Contexts: A clear boundary within which a specific domain model applies. The definition of a term can change completely depending on the context. For instance, in ProjectForge:
    • In the **Collaboration Context**, a Member represents a user collaborating on tasks.
    • In the **Billing Context**, a Member represents a seat license on an invoice.
    Trying to create a single global User model that handles both collaboration and billing details is an anti-pattern. They must be split into separate models within their respective Bounded Contexts.

Domain Model Building Blocks

Inside a Bounded Context, we organize domain objects into three primary structural blocks:

1. Entities

Objects that have a unique thread of identity that persists over time, despite changes to their internal attributes. They are not defined by their values, but by their ID.

ProjectForge Example: A Task is an Entity. It has a unique id. Its status can change from TODO to DONE, its assignee can change, and its title can be edited, but it remains the exact same task.

2. Value Objects (VO)

Objects that have no identity of their own. They are defined entirely by their attributes and values. Value Objects are immutable. If you want to change a Value Object, you replace the entire instance.

ProjectForge Example: TaskPriority (representing a priority like `URGENT` or `LOW`) or a TaskLabel (with attributes name: "Bug" and color: "#ff0000"). If you change the color, you don't mutate the object; you construct a new TaskLabel value.

3. Aggregates & Aggregate Roots

An **Aggregate** is a cluster of associated Entities and Value Objects that are treated as a single transactional unit for data changes. The **Aggregate Root** is the gatekeeper Entity that controls all access to the internal objects of the Aggregate. External objects can only reference the Aggregate Root.

ProjectForge Example: A Project acts as an Aggregate Root. It manages child entities like Task and value objects like TaskLabel. You cannot modify a Task directly without going through the Project aggregate root, which enforces consistency rules (e.g., "A task cannot be marked completed if the project is archived").

Quiz: Test Your Storage Strength
1. In ProjectForge, what makes a Project an Entity rather than a Value Object?
Correct! An Entity is defined by a unique identity key that remains unchanged over time, whereas Value Objects are defined solely by the values of their attributes.
Incorrect. Recall the definition of an Entity. While Projects manage tasks and belong to workspaces, the primary trait that distinguishes an Entity from a Value Object is how it is identified and tracked over time.
2. What is the primary role of an Aggregate Root in a Bounded Context?
Correct! The Aggregate Root governs transactional consistency boundaries, ensuring that all invariant business rules are satisfied whenever data within the Aggregate is updated.
Incorrect. Think about data isolation and invariants. While it encapsulates database fields, its primary design role in DDD is acting as the single transactional gatekeeper to maintain consistency rules.
Desirable Difficulty Tip: When designing models in TypeScript, you can enforce Value Object immutability by using the readonly keyword. For example: 
interface TaskLabel {
  readonly name: string;
  readonly color: string;
}
Attempting to mutate a label's color will result in a compiler error, forcing you to create a fresh Value Object instance.
