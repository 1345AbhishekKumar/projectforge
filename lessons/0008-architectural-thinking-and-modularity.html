Lesson 0008 • Modularity

Architectural Thinking & Modularity

As applications grow from simple prototypes to enterprise platforms, their internal boundaries must evolve. Architectural thinking is the discipline of structuring code systems to control complexity, isolate change, and establish scalable boundaries.

The Core Principles: Cohesion and Coupling

Every software architecture can be evaluated by how well it balances two core metrics:

  • Cohesion (High is Good): How closely related the responsibilities inside a single module or file are. A module has high cohesion if all its code works together to solve a single, well-defined problem (e.g., a `Billing` module that only does invoice math, tax calculation, and payment processing).
  • Coupling (Low is Good): The strength of the dependencies between different modules. Two modules are highly coupled if a change in one requires code modifications in the other (e.g., if changing the schema of the `Task` table breaks the billing invoice layout, the modules are tightly coupled).

The Architectural Evolution Spectrum

Architectures are not static; they represent a spectrum of tradeoffs. For ProjectForge, this spectrum maps to three primary stages:

1. Monolithic Architecture

All application code (UI, server logic, database scripts) lives in a single project and deploys as a single executable. Next.js App Router projects often start here, where Server Actions connect directly to database queries inside page files.

Trade-off: High speed of initial development, but poor boundary control. Over time, it is easy to write spaghetti dependencies between features.

2. Modular Monolith

The application still deploys as a single unit, but the codebase is strictly segregated into independent directories (modules) representing bounded contexts (e.g., `/modules/collaboration`, `/modules/billing`). Inter-module communication is restricted to clear API boundaries or event buses.

Trade-off: Prevents spaghetti dependencies while avoiding the operational overhead (deployment pipelines, networks) of microservices.

3. Microservices Architecture

Each Bounded Context is split into a completely separate project, running on separate server hosts and maintaining its own isolated database (Database-per-Service pattern). Communication occurs asynchronously over HTTP APIs or event brokers (like Kafka or RabbitMQ).

Trade-off: Highly scalable and independent deployment, but extremely complex to debug, deploy, and maintain. Transactional consistency becomes difficult (requiring patterns like Saga Orchestration).

Database Boundaries in Serverless Next.js

In a Next.js environment like ProjectForge, database connection pooling limits (especially in serverless environments) represent a critical constraint. Tightly coupling multiple serverless functions to a single Postgres database instance can lead to pool exhaustion.

By enforcing a **Modular Monolith** boundary early, you can design your database schemas to be decoupled. If the `Collaboration` schema and the `Billing` schema do not perform SQL joins across boundaries, they can easily be split into separate databases when transitioning to microservices later.

Interactive Simulator: Code Design Decisions

Evaluate the impact of the following architectural design choices on **ProjectForge**. Categorize them as **High Cohesion (HC)**, **Low Coupling (LC)**, or **Tight Coupling (TC)**.

"Executing direct SQL joins between task collaboration tables and billing invoice tables."
"Exposing an asynchronous Event Bus to notify Billing when a new user is invited."
"Grouping all invoice formatting, tax calculations, and discount rules in a single module."
"Allowing UI pages to directly access database client pools and modify user records."
Superb logic! Direct SQL cross-joins or client pool mutations introduce tight coupling (making it hard to refactor). Restricting cross-boundary communication to an event bus ensures low coupling, while keeping related billing rules unified ensures high cohesion.

Correct! Low coupling minimizes dependency chains, ensuring that refactoring or updating logic inside one module does not break or disrupt separate systems.
Incorrect. Recall what coupling measures. Highly coupled modules break together. What do we achieve when we decouple them? Choose the option related to isolation of changes.

Correct! Isolating backend/server logic inside decoupled modules or folders within a single project enforces the logical boundaries of a modular monolith before needing physical deployments.
Incorrect. A modular monolith deploys as a single application, so separate cloud hosts represent microservices. What design boundary isolates code logic within a single codebase?
Desirable Difficulty Tip: When importing code from other modules in a modular monolith, avoid reaching deep into folders (e.g., import { helper } from "../../billing/utils/private/helper"). Instead, export a public API at the root of the module (an `index.ts` file) and restrict imports to that public interface.
