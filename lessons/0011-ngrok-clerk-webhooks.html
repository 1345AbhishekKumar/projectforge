Hooking up ngrok with Clerk Webhooks

Bridging the gap between external identity systems and your local database.

In our last lesson, we learned that ngrok exposes your localhost to the internet. Today, we look at the engineering reality of using this tunnel to sync data between a SaaS Auth provider (Clerk) and your application (ProjectForge).

The Synchronization Problem

If you use Clerk for authentication, Clerk owns the User Identity (email, passwords, OAuth tokens). But ProjectForge owns the Domain Data (e.g., projects, tasks, user profiles). When a user signs up on Clerk's hosted page, your database doesn't magically know about it.

To sync them, we use Webhooks: Clerk sends an HTTP POST request to your API every time a user is created, updated, or deleted.

The Development Workflow

  1. Start your dev server: npm run dev (runs on port 3000).
  2. Start ngrok: ngrok http 3000. ngrok gives you a forwarding URL like https://ab12-cd34.ngrok-free.app.
  3. Configure Clerk: Go to the Clerk Dashboard -> Webhooks. Add an endpoint pointing to https://ab12-cd34.ngrok-free.app/api/webhooks/clerk. Choose the events you care about (e.g., user.created).
  4. Code the Endpoint: In Next.js, you create a route handler at app/api/webhooks/clerk/route.ts to accept this POST request and insert a new user row into your PostgreSQL database.

The Critical Threat: Webhook Forgery

Think like an engineer. You just exposed /api/webhooks/clerk to the public internet using ngrok. What stops a malicious actor from discovering that URL and sending a fake HTTP POST request that says "Create a new admin user" or "Delete user ID 5"?

Nothing.

Because HTTP is stateless and anyone can send a POST request, you must verify the authenticity of the request. You need mathematical proof that the request definitely came from Clerk and that the payload hasn't been tampered with in transit.

Cryptographic Signatures (Svix)

Clerk uses a service called Svix to send webhooks. Here is how it protects your endpoint:

  • When you create a webhook in Clerk, you get a Signing Secret. You store this securely in your .env file (e.g., WEBHOOK_SECRET=whsec_...). This secret is known only to Clerk and your server.
  • When Clerk sends the webhook, it calculates a cryptographic hash (HMAC) of the payload using that Signing Secret. It attaches this hash in the HTTP headers (e.g., svix-signature).
  • When your Next.js route receives the request, it takes the incoming payload and your local WEBHOOK_SECRET, and calculates the hash itself.
  • If your calculated hash matches the hash in the header, the request is authentic. If it doesn't, you reject the request with a 400 Bad Request.

By verifying the signature, it doesn't matter if ngrok exposes your port to the entire internet—only requests cryptographically signed by Clerk will be processed.
