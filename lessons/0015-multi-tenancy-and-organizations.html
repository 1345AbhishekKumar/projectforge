Multi-Tenancy & Organizations

Transitioning from B2C logic to scalable B2B Architectures.

Until now, we have assumed a B2C (Business to Consumer) model. A user signs up, logs in, and views their data. The core identity is the User. But most SaaS applications, including ProjectForge, are B2B (Business to Business). In B2B, the core entity is not the User; it is the Organization (or Workspace, Team, Tenant).

This architectural shift is called Multi-Tenancy. A single instance of your software serves multiple distinct tenants (organizations). A user does not own a project; an Organization owns the project, and users are Members of that Organization with specific roles.

1. The Architectural Shift: Data Isolation

In B2C, your database queries look like this:

// B2C Query: Fetch my projects
SELECT * FROM projects WHERE user_id = 'user_123';

In a B2B multi-tenant application, users can belong to multiple organizations. When they are logged into the "Acme Corp" context, they should absolutely NOT see data from "Globex Inc". Data isolation is the most critical security boundary in SaaS.

// B2B Query: Fetch projects for the currently active organization
SELECT * FROM projects WHERE org_id = 'org_abc';

The Cross-Tenant Data Leak

If you fail to include WHERE org_id = ? on a query in a B2B app, you will leak proprietary data from one company to another. This is a catastrophic security failure that destroys business trust instantly. Every domain model in a B2B app must associate with a Tenant ID.

2. Modeling Multi-Tenancy in the Database

Because we use Clerk for organizations, Clerk handles the complexities of invitations, billing limits on seats, and role assignments (Admin vs Member). However, just like we synced Users in Lesson 13, we must sync Organizations to maintain referential integrity in our local database.

// schema.prisma

model Organization {
  id        String   @id @unique // Stores the Clerk org_id
  name      String
  slug      String?  @unique
  imageUrl  String?

  // The Organization owns the domain data
  projects  Project[]
  members   OrgMembership[]
}

// The Join Table: Users can be in many Orgs, Orgs have many Users
model OrgMembership {
  id        String       @id @default(cuid())
  role      String       // e.g., 'org:admin', 'org:member'
  
  userId    String
  user      User         @relation(fields: [userId], references: [id])
  
  orgId     String
  org       Organization @relation(fields: [orgId], references: [id])

  @@unique([userId, orgId]) // A user has only one membership per org
}

model Project {
  id        String       @id @default(cuid())
  title     String
  
  // Bound to the Organization, NOT the User
  orgId     String
  org       Organization @relation(fields: [orgId], references: [id])
}

To keep this in sync, you would expand your Webhook handler from Lesson 13 to listen to organization.created, organization.updated, and organizationMembership.created events.

3. Routing by Organization (Next.js Middleware)

In a B2B app, URLs often encode the organization context, e.g., /orgs/acme-corp/projects. But even if they don't, your backend needs to know which organization the user is currently operating as.

Clerk introduces the concept of an Active Organization. A user might belong to 5 orgs, but they are only "acting" as one at any given time. Clerk's token provides the orgId of the currently active organization.

We can use Middleware to force users to select an organization before they can access the application.

// middleware.ts
import { clerkMiddleware, createRouteMatcher } from "@clerk/nextjs/server";
import { NextResponse } from "next/server";

const isPublicRoute = createRouteMatcher(['/', '/sign-in(.*)', '/sign-up(.*)']);

export default clerkMiddleware((auth, req) => {
  const { userId, orgId } = auth();

  // 1. Protect all non-public routes
  if (!isPublicRoute(req)) {
    auth().protect();
  }

  // 2. Enforce Organization Selection
  // If they are logged in, but haven't selected an active organization,
  // and they are NOT already on the org-selection page, redirect them.
  if (userId && !orgId && req.nextUrl.pathname !== '/org-selection') {
    const orgSelectionUrl = new URL('/org-selection', req.url);
    return NextResponse.redirect(orgSelectionUrl);
  }
});

UX Considerations: The Orphaned User

What happens if a user signs up, but they haven't been invited to any organization and haven't created one? The middleware above will trap them on /org-selection. Your /org-selection page must gracefully handle this state by offering a "Create an Organization" form or a message instructing them to wait for an email invitation.

4. Context in Server Components

Once the middleware guarantees the user has an active organization, accessing that context in a Next.js Server Component is trivial and highly secure.

// app/dashboard/projects/page.tsx (Server Component)
import { auth } from '@clerk/nextjs/server';
import { db } from '@/lib/db';

export default async function ProjectsPage() {
  // The middleware ensures orgId exists, but TS requires a fallback
  const { orgId } = auth();
  if (!orgId) return <div>Unauthorized</div>;

  // B2B Data Isolation enforced at the query level!
  const projects = await db.project.findMany({
    where: { orgId: orgId }
  });

  return (
    <div>
      <h1>Organization Projects</h1>
      <ul>
        {projects.map(p => <li key={p.id}>{p.title}</li>)}
      </ul>
    </div>
  );
}

5. Clerk's Drop-in Organization Components

Building organization management UI (inviting members, managing roles, switching orgs) from scratch is incredibly tedious. Clerk provides drop-in Client Components that handle all of this safely.

'use client';
import { OrganizationSwitcher, OrganizationProfile } from '@clerk/nextjs';

export function Navbar() {
  return (
    <nav>
      {/* Allows the user to toggle between their organizations */}
      <OrganizationSwitcher hidePersonal={true} />
    </nav>
  );
}

export function OrgSettingsPage() {
  return (
    <div>
      {/* Fully featured UI for inviting users and managing roles */}
      <OrganizationProfile />
    </div>
  );
}
