Advanced RBAC & Granular Permissions

Moving beyond "Who are you?" to "What exactly are you allowed to do?"

In Lesson 15, we learned how to architect Multi-Tenancy. We successfully isolated data so that an employee at Acme Corp cannot view data belonging to Globex Inc. But what happens within Acme Corp?

Should an intern at Acme Corp have the ability to delete the entire corporate workspace? Should a guest contractor be able to view the billing details? Absolutely not. This brings us to Role-Based Access Control (RBAC) and Granular Permissions.

1. The Anti-Pattern: Hardcoding Roles

When engineers first implement authorization, they often write code like this:

// ❌ The Anti-Pattern: Checking against hardcoded role strings
if (user.role === 'admin') {
  showDeleteButton();
}

This seems fine initially, but as the company scales, product management asks for a new role: manager. Managers should also be able to delete projects. So, the engineer updates the code:

// ❌ The scaling nightmare begins
if (user.role === 'admin' || user.role === 'manager') {
  showDeleteButton();
}

Two months later, enterprise customers demand custom roles. They want a billing_admin who can manage subscriptions but cannot delete projects. Your entire codebase is now littered with fragile if/else role-checking logic that is impossible to maintain.

2. The Solution: Granular Permissions

The golden rule of scalable authorization is: Assign Permissions to Roles, and Roles to Users. Then, evaluate Permissions in your code, NEVER Roles.

Clerk handles this beautifully. In the Clerk Dashboard, you define Roles (e.g., `org:admin`, `org:member`). Then, you define granular Permissions (e.g., `org:projects:delete`, `org:billing:manage`) and assign them to those Roles.

Your codebase then becomes completely decoupled from specific role names. You only ask: "Does this session have the permission to do X?"

// ✅ The Scalable Pattern: Checking Permissions
const { has } = auth();

// 'has()' checks the user's active organization permissions
if (has({ permission: 'org:projects:delete' })) {
  showDeleteButton();
}

Future-Proof Architecture

If a customer requests a custom role tomorrow, you simply configure it in the Clerk Dashboard and give it the `org:projects:delete` permission. You do not need to change a single line of code or deploy a new version of your application.

3. Enforcing Permissions in Next.js

Just like general authentication, permissions must be enforced at multiple layers of your stack to ensure both security and a good User Experience (UX).

A. Server Components (Security Boundary)

In a Server Component or Server Action, verifying permissions is mandatory. This is your backend security boundary. If a malicious user bypasses the UI and sends an HTTP request directly to a Server Action, this check blocks them.

// app/dashboard/projects/[id]/page.tsx
import { auth } from '@clerk/nextjs/server';
import { redirect } from 'next/navigation';

export default async function ProjectSettings() {
  const { has } = auth();

  // Strictly enforce the permission on the server
  const canManageSettings = has({ permission: 'org:projects:manage_settings' });

  if (!canManageSettings) {
    // Redirect unauthorized users away, or return a generic Error Component
    redirect('/unauthorized');
  }

  return (
    <div>
      <h1>Danger Zone</h1>
      <DeleteProjectButton />
    </div>
  );
}

B. Client Components (UX Boundary)

While the server enforces security, the client enforces UX. You should not show a "Delete" button to a user who doesn't have the permission to click it. That creates frustration.

In Client Components, we use the useAuth() hook.

'use client';
import { useAuth } from '@clerk/nextjs';

export function ProjectToolbar() {
  const { isLoaded, has } = useAuth();

  if (!isLoaded) return <SkeletonLoader />;

  const canDelete = has({ permission: 'org:projects:delete' });

  return (
    <div>
      <button>Edit Project</button>
      
      {/* Conditionally render the destructive action */}
      {canDelete && (
        <button className="bg-red-500">Delete Project</button>
      )}
    </div>
  );
}

UX Best Practice: Hide vs Disable

Should you hide the button entirely, or show it as disabled? Hide it if the user has no business knowing the feature exists (e.g., advanced billing settings). Disable it if the feature is a core part of the app, and you want them to know they lack access (often accompanied by a tooltip: "Contact your Admin to access this feature").

4. Enforcing Permissions at the Edge (Middleware)

For entire routes or sections of your application, you can prevent access at the Edge using Middleware, saving Serverless function execution time and database reads.

// middleware.ts
import { clerkMiddleware, createRouteMatcher } from "@clerk/nextjs/server";
import { NextResponse } from "next/server";

const isBillingRoute = createRouteMatcher(['/dashboard/billing(.*)']);

export default clerkMiddleware((auth, req) => {
  const authObj = auth();

  // Intercept requests to the billing section
  if (isBillingRoute(req)) {
    const canManageBilling = authObj.has({ permission: 'org:billing:manage' });
    
    if (!canManageBilling) {
      // Halt the request at the edge and redirect
      const url = new URL('/unauthorized', req.url);
      return NextResponse.redirect(url);
    }
  }
});
