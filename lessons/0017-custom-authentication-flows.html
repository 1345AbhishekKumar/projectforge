Custom Authentication Flows

Taking absolute control of the user onboarding experience with headless auth.

In our previous lessons, we successfully orchestrated B2B logic, Webhooks, Data Isolation, and Granular Permissions. Throughout these examples, we implicitly relied on Clerk's drop-in <SignIn /> and <SignUp /> components. These components are excellent for getting to market quickly.

However, as ProjectForge matures into an Enterprise SaaS, design constraints emerge. You might need a complex multi-step onboarding wizard, highly customized CSS that defies pre-built component structures, or you might want to embed the sign-in form directly into a promotional landing page. This requires dropping the UI components and utilizing Headless Authentication.

1. Headless Authentication: The Concept

Headless authentication means you are building 100% of the UI (the HTML inputs, buttons, and loading spinners) and wiring them up to an authentication API via React Hooks. You handle the state, the provider handles the cryptography and session management.

In Clerk, the headless engine is powered by two primary React hooks:

  • useSignIn(): Manages the state machine for returning users.
  • useSignUp(): Manages the state machine for new users.

2. The Authentication State Machine

Authentication is not a simple "Success/Fail" binary. It is a State Machine. When a user submits an email and password, the request might succeed, but the authentication flow might not be "Complete".

Consider a user with Two-Factor Authentication (2FA) enabled. Submitting the password transitions the state machine to needs_second_factor. You must then prompt them for an SMS code or Authenticator App code. Only after that succeeds does the state transition to complete.

The signIn.status property drives your UI:

  • null: The flow hasn't started.
  • needs_identifier: Waiting for an email or username.
  • needs_first_factor: Waiting for a password or email link click.
  • needs_second_factor: Waiting for a 2FA code.
  • complete: The session is established. The user is logged in.

3. Building the Custom Sign-In Component

Let's build a bespoke email and password sign-in form in Next.js using TailwindCSS styling principles. We must handle the form state locally, submit it to Clerk, and interpret the resulting status.

'use client';

import { useState } from 'react';
import { useSignIn } from '@clerk/nextjs';
import { useRouter } from 'next/navigation';

export default function CustomSignIn() {
  const { isLoaded, signIn, setActive } = useSignIn();
  const router = useRouter();

  // Local UI state
  const [emailAddress, setEmailAddress] = useState('');
  const [password, setPassword] = useState('');
  const [error, setError] = useState('');

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    if (!isLoaded) return;

    try {
      // 1. Start the Sign In process with the provided credentials
      const result = await signIn.create({
        identifier: emailAddress,
        password,
      });

      // 2. Check the State Machine
      if (result.status === 'complete') {
        // 3a. Success! Establish the session and redirect
        await setActive({ session: result.createdSessionId });
        router.push('/dashboard');
      } else {
        // 3b. The user requires 2FA or further action
        // In a real app, you would redirect to a '/verify-2fa' page
        console.log(result);
        setError('Additional verification required.');
      }
    } catch (err: any) {
      // 4. Handle incorrect passwords or unknown users
      console.error('Error signing in:', err);
      setError(err.errors[0]?.message || 'Invalid credentials');
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <h2>Sign in to ProjectForge</h2>
      {error && <p style={{color: 'red'}}>{error}</p>}
      
      <input 
        type="email" 
        value={emailAddress} 
        onChange={(e) => setEmailAddress(e.target.value)} 
        placeholder="Email"
      />
      <input 
        type="password" 
        value={password} 
        onChange={(e) => setPassword(e.target.value)} 
        placeholder="Password"
      />
      <button type="submit">Sign In</button>
    </form>
  );
}

The Critical setActive() Call

A common mistake when building headless auth is forgetting to call setActive({ session: result.createdSessionId }). Even if the status is complete, the user is not technically "logged in" to your frontend application until you pass the newly generated session ID to the setActive function. This function creates the actual JWT cookie in the browser that allows middleware and server components to recognize the user.

4. Advanced Flow: Email Verification (OTP)

Modern applications increasingly rely on Passwordless authentication (One-Time Passwords sent to email). In this scenario, the user inputs their email, and the system transitions to needs_first_factor. You then show an input for the 6-digit code.

// Step 1: Send the OTP
const startEmailOtp = async () => {
  await signIn.create({
    strategy: "email_code",
    identifier: emailAddress,
  });
  // Update UI state to show the Code Input field
  setShowCodeInput(true); 
};

// Step 2: Verify the OTP
const verifyCode = async (code: string) => {
  const result = await signIn.attemptFirstFactor({
    strategy: "email_code",
    code: code,
  });

  if (result.status === 'complete') {
    await setActive({ session: result.createdSessionId });
    router.push('/dashboard');
  }
};

By breaking the process down into discrete, state-driven steps via the useSignIn hook, you have absolute freedom to render any UI, handle any error message, and design complex split-testing flows for onboarding.
