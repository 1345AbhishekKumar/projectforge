Lesson 0020: Row Level Security (RLS)

In previous lessons, we successfully established identity using Clerk, securely synced user data to our database via Webhooks, and routed it locally with Ngrok. But once a user is in your database, how do you prevent them from accessing another user's data?

The Old Way: Application-Level Authorization

Traditionally, developers handle data ownership entirely within the application backend (e.g., Node.js, Python). When querying the database, you explicitly tell it to filter by the current user's ID:

// Application-level authorization check
const currentUserId = req.user.id;
const userProjects = await db.query(
  'SELECT * FROM projects WHERE user_id = $1', 
  [currentUserId]
);

The problem: If you forget to add that WHERE user_id = $1 clause anywhere in your application, you have instantly created a catastrophic data leak. Your API endpoint might return every project from every user.

The New Way: Row Level Security (RLS)

Row Level Security (RLS) is a database feature (popularized by PostgreSQL) that pushes the authorization logic down to the database itself. Instead of trusting your application code, the database inspects every query and automatically filters out rows the user isn't allowed to see.

Implementing RLS usually involves two simple steps:

  1. Enable RLS on your table:
    ALTER TABLE projects ENABLE ROW LEVEL SECURITY;
  2. Create a Policy (The Rule):
    You write an expression that evaluates to true or false for each row.
CREATE POLICY "Users can view their own projects" 
ON projects 
FOR SELECT 
USING (user_id = auth.uid());

In this PostgreSQL example, auth.uid() extracts the user ID from the database session. Now, even if a junior developer writes SELECT * FROM projects in the backend, the database intercepts it, applies the policy, and only returns that user's projects.

Knowledge Check

If you execute SELECT * FROM projects on a table that has RLS enabled and a policy restricting reads to user_id = auth.uid(), what will happen if an authenticated user runs this query?
