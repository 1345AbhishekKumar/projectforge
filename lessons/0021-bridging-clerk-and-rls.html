Lesson 0021: Bridging Clerk & RLS

In Lesson 20, we learned that RLS acts as a "silent filter" in the database. But the database is isolated. It doesn't know who is currently logged into your Next.js app. It doesn't know about Clerk.

To make RLS work, we must securely pass the User ID from your Backend (Next.js) to the Database (PostgreSQL) for every single request.

The Mechanism: SET LOCAL

PostgreSQL has a feature called "Config Settings" (GUCs). You can set custom variables that exist only for the duration of a single Transaction.

In your Next.js Server Action or API Route, you perform these steps:

  1. Get the userId from Clerk using auth().
  2. Open a database transaction.
  3. Run a SET LOCAL command to store that ID.
  4. Execute your actual query.
// Inside your Next.js Server Action
import { auth } from "@clerk/nextjs/server";

export async function getProjects() {
  const { userId } = await auth();
  
  return await db.transaction(async (tx) => {
    // 1. Tell Postgres who the user is
    await tx.execute(
      sql`SELECT set_config('app.current_user_id', ${userId}, true)`
    );

    // 2. Run the query - RLS will now use 'app.current_user_id'
    return await tx.select().from(projects);
  });
}

The true in set_config(..., true) means the setting is local to the transaction. Once the transaction finishes, the setting is wiped clean. This prevents "Identity Leakage" between different web requests using the same database connection pool.

The RLS Policy Update

Now that the variable app.current_user_id is set, your RLS policy needs to read it:

CREATE POLICY "User Project Access" 
ON projects 
USING (
  user_id = current_setting('app.current_user_id')
);
Security Note: We use a transaction because if we set it globally on the connection, another user's request might "inherit" the previous identity before it's overwritten!

Knowledge Check

Why is it critical to use SET LOCAL or set_config(..., true) instead of a global configuration change when passing the user ID?
