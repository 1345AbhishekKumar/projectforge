Lesson 0022: Advanced RLS Patterns

We know how to enable RLS and how to bridge it with Clerk. Now, we move into Production-Grade Engineering. In a real SaaS, security isn't just about "my user ID." It's about organizations, roles, performance, and bulletproof data integrity across all CRUD operations.

1. The CRUD Nuance: USING vs WITH CHECK

Most beginners only use USING. But for INSERT and UPDATE, you need to know the difference. If you don't, a user might be able to "steal" a row by changing its user_id to their own, or "abandon" a row by changing its ID to someone else's.

USING clause

Applies to rows already in the table. It determines if you can see or select the row.

WITH CHECK clause

Applies to rows being created or modified. It determines if the new state of the row is valid.

The "Bulletproof" UPDATE Policy

When a user updates a project, we must ensure:
1. They own the project currently (USING).
2. They aren't trying to change the user_id to a different user (WITH CHECK). 

CREATE POLICY "Users can update their own projects"
ON projects
FOR UPDATE
USING (user_id = current_setting('app.current_user_id'))
WITH CHECK (user_id = current_setting('app.current_user_id'));

If you omit WITH CHECK, a user could run UPDATE projects SET user_id = 'hacker_id' WHERE id = 123. The USING clause would let them find the row (because they own it), and the database would let the update happen, effectively "gifting" the row to another user!

2. Multi-Tenancy: Organization-Level RLS

In a SaaS like ProjectForge, users belong to Organizations. A project isn't owned by a user; it's owned by an org. Multiple users in the same org should see the same projects.

The Schema Pattern

Every table must have an org_id. Your Clerk JWT should also include the org_id in its claims.

-- 1. Create the policy for Org-based access
CREATE POLICY "Org members can view projects"
ON projects
FOR SELECT
USING (
  org_id = current_setting('app.current_org_id')
);

-- 2. Next.js Transaction Bridge
await tx.execute(sql`
  SELECT set_config('app.current_user_id', ${userId}, true);
  SELECT set_config('app.current_org_id', ${orgId}, true);
`);

3. RBAC (Role-Based Access Control) in RLS

What if only "Admins" in an organization can delete projects, but "Members" can only view them? We can bake roles directly into the RLS policy.

CREATE POLICY "Only admins can delete projects"
ON projects
FOR DELETE
USING (
  org_id = current_setting('app.current_org_id') 
  AND 
  current_setting('app.user_role') = 'admin'
);

This is extremely powerful. Your application code just tries to run DELETE FROM projects WHERE id = 1. If the user is a 'member', the database returns 0 rows affected. No complex if/else logic in your frontend or backend!

🚀 Performance Tip: Avoid Subqueries

A common mistake is writing policies that query other tables:

-- BAD PERFORMANCE
USING (
  EXISTS (SELECT 1 FROM memberships WHERE user_id = auth.uid() AND org_id = projects.org_id)
)

Why this is slow: This subquery runs for every single row in the projects table during a SELECT *. If you have 10,000 projects, you just ran 10,000 subqueries!

The Fix: Pass the necessary context (like org_id or role) directly into the database session via set_config. Trust your JWT/Clerk session to provide these values, so the RLS policy only has to compare two simple strings.

4. Bypassing RLS for System Tasks

Sometimes your system needs to do background work (e.g., a cron job or a webhook sync) that shouldn't be restricted by RLS. You have two options:

  • The Service Role: Use a database user that is a SUPERUSER or has BYPASSRLS attribute.
  • The Admin Override: In a specific transaction, you can SET row_security = off; (requires high privileges).

Best Practice: Use a separate "service" connection string for background workers and a "user" connection string for your Next.js API routes.

Knowledge Check: The Ultimate RLS Test

You have a table tasks with RLS enabled. You have a policy for UPDATE that has a USING clause but no WITH CHECK clause. A user who owns a task tries to update that task and change its owner_id to their friend's ID. What happens?
